# Database Schema (ERD) Strategy
*Note: Due to the massive scale of the ERP, this highlights the core structure required for multi-tenancy and the domains requested.*

## Key ID Rules (Non-Negotiable)
- Use standard massive integer or UUIDs. However, sequential IDs are needed as per requirement.
- To prevent conflicts, Composite Primary Keys or Global Unique Identifiers must be used.
- Example: `id` (Auto Increment / UUID) is PK, but business referential IDs like `order_number` are strictly generated by format: `{ClientID}-{BranchID}-{CounterID}-{Sequence}`.

## 1. Global Platform Models

**Table: `clients` (Clients)**
- `id` (PK, Auto Increment)
- `company_name`
- `domain_slug`
- `status` (Active, Suspended, Expired, Read-only)
- `subscription_plan_id`
- `expiry_date`
- `grace_period_days`
- `created_at`, `updated_at`

**Table: `system_users` (Super Admins)**
- `id`
- `email`, `password_hash`, `role`

**Table: `subscription_plans`**
- `id`
- `plan_code`
- `plan_name`
- `monthly_price`, `annual_price`
- `max_branches`, `max_users`
- `allowed_modules` (JSON array: "pos, inventory, finance")

**Table: `client_subscriptions`**
- `id`
- `client_id` (FK)
- `plan_id` (FK)
- `billing_cycle` (`monthly`, `annual`)
- `subscription_status`
- `price_snapshot`
- `effective_start_at`, `effective_end_at`

## 2. Master & Setup (Client Side) - `client_id` isolated

**Table: `branches`**
- `id` (PK)
- `client_id` (FK) -> Indexes with `branch_id`
- `branch_code` (Unique per client)
- `branch_name`, `address`, `tax_region`

**Table: `users`**
- `id` (PK)
- `client_id` (FK)
- `employee_id` (Generated: ClientID + Sequential)
- `branch_id` (Nullable for global client admins)
- `user_name`, `password_hash`, `pin_code` (for POS)
- `role_id` 

**Table: `roles` & `permissions`**
- `id` (PK)
- `role_name`
- Custom active-directory style groups for exact feature access.

## 3. Product Builder Module

**Table: `products`**
- `id`, `client_id`, `category_id`
- `product_name`, `sku`, `base_price`
- `menu_type` (Ala carte, buffet, weekly)
- `is_configurable` (Boolean)

**Table: `product_configurations` (Pizza size, flavor, dough, etc.)**
- `id`, `product_id`
- `attribute_type` (Size, Flavor, Topping, Crust, Sauce)
- `attribute_value` (Large, BBQ, Cheese, Pan, Marinara)
- `price_modifier` (+2.00)

## 4. POS & Ordering

**Table: `orders`**
- `id` (UUID generated on POS perfectly to avoid offline clashes)
- `client_id`, `branch_id`, `counter_id`
- `order_number` (String: `[Client]-[Branch]-[Counter]-[Sequence]`)
- `status` (Open, Paid, Cancelled, Void)
- `customer_id` (Nullable)
- `order_type` (Dine-in, Takeaway, Delivery)
- `total_amount`, `tax_amount`, `discount_amount`
- `sync_status` (Pending, Synced)

**Table: `order_items`**
- `id`, `order_id`
- `product_id`, `qty`, `unit_price`, `total`
- `configuration_json` (snapshot of sizing/toppings)
- `kot_number` (e.g., KOT-235-1)
- `kot_status` (Pending, Cooking, Ready, Served)

## 5. Finance & Accounting (Double Entry)

**Table: `chart_of_accounts`**
- `id`, `client_id`
- `account_code`, `account_name`
- `account_type` (Asset, Liability, Equity, Revenue, Expense)
- `parent_account_id`

**Table: `general_ledger` (Journal Entries)**
- `id`, `client_id`
- `transaction_date`, `reference` (e.g., Order ID, Voucher)
- `account_id`
- `debit_amount`, `credit_amount`
- `currency_code`, `exchange_rate`

## 6. Governance & Audit Logging
**Table: `audit_logs`**
- `id`, `client_id`
- `user_id` (Actor)
- `action` (CREATE, UPDATE, DELETE, OVERRIDE)
- `entity_type` (Order, Product, Inventory)
- `entity_id`
- `old_values` (JSON)
- `new_values` (JSON)
- `reason` (String - for overrides)
- `timestamp`
