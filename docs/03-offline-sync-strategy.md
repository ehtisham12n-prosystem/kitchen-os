# Offline POS Sync Strategy & Conflict Resolution

## 1. The Offline-First Challenge
The Windows POS must function flawlessly with 0% internet. This means it must:
1. Create Orders, Payments, Customers.
2. Generate absolutely unique, sequential IDs that do not clash.
3. Manage KOT numbers that reset daily without a central server orchestrating it.
4. Not edit central entities (like Products or Accounts) unless online.

## 2. ID Generation Rules
To prevent collisions between multiple offline terminals:

### UUID for Database Primary Keys
Every row strictly gets a `UUID` v4 at the moment of creation on the POS. This guarantees no row-level database conflicts during sync.

### Deterministic Format for Business Counters (Order Number)
Users need readable Order Numbers (not UUIDs).
Format: `[Client ID]-[Branch ID]-[Counter ID]-[Sequential Local ID]`
- **Client ID**: 001
- **Branch ID**: B1
- **Counter ID**: POS1
- **Sequence**: 00045 (Tracked locally in the SQLite DB of that specific POS)
Result: `001-B1-POS1-00045`

Because `Counter ID` is uniquely assigned to the terminal, **no two terminals offline can ever generate the same Order Number**.

### KOT Number Strategy
Requirement: "Resets daily per branch. Versioned on modification: KOT-235, KOT-235-1."
- **Locally**: Each POS operates KOT counters. Wait, if multiple POS systems share a kitchen, they need consecutive KOT numbers.
- **Offline LAN setup**: The POS terminals must talk to a **Local Branch Master** (usually the main POS or a local server machine) over LAN. KOT sequence is generated by the Local Branch Master. 
- **If completely isolated (Tablet mode)**: KOT is formatted as `POS1-KOT-235`.

## 3. Data Sync Flow (Bi-Directional)

### POS -> Cloud (Uplink)
- **Trigger**: Network detection goes online, or background worker checks every 5 seconds.
- **Payload**: Pushes all local SQLite rows where `sync_status = pending`.
- **Batching**: Transactions are batched. For orders, the `Order`, `Order_Items`, `Payments`, and `GL_Entries` are pushed in a single atomic payload. 
- **Conflict Resolution**: "Cloud wins" on master data, but "POS wins" on transactional data since it's the source of truth for the offline order.

### Cloud -> POS (Downlink)
- **Trigger**: Web-socket push when prices, products, or users change in the Cloud.
- **Scope**: Subscribes only to its `branch_id` and global `client_id` data.
- **Safety**: POS will not apply changes to an item that is currently in an active open order.

## 4. Preventing Modification of Synced Data
If a closed order is synced, and the manager needs to Void it (Offline):
- The POS creates a **Void Event** (Compensating Transaction) with an override reason. 
- It does NOT mutate the original row; it appends a modification log and a reversing double-entry line. 
- This ensures full auditability when synced.
